# AI_SYS 프로젝트 진행 현황 및 향후 실행 로드맵

작성일: 2026-04-23
기준 브랜치/저장소 상태: 현재 워크스페이스 코드 기준
문서 목적: 지금까지 구현된 범위를 명확히 정리하고, 남은 작업을 우선순위/실행 단위로 구체화

---

## 1. 프로젝트 한 줄 요약

AI_SYS는 경찰 공무원 시험 준비를 위한 판례 학습 플랫폼으로, iOS 앱에서 OCR/검색/요약/퀴즈/오답복습 흐름을 제공하고 FastAPI + PostgreSQL 백엔드와 연동해 근거 기반 학습을 지원한다.

---

## 2. 현재까지 진행사항 (완료/구현 상태)

### 2-1. 전체 아키텍처/인프라

- 완료: iOS 앱(SwiftUI) - FastAPI - PostgreSQL(pgvector 포함 스키마) 기본 연결 구조 확립
- 완료: Docker Compose 기반 API/DB 동시 실행 경로 정리
- 완료: API 상태 점검용 `/health` 엔드포인트와 문서(`/docs`) 기반 확인 루틴 마련
- 완료: 로컬 실행 가이드 및 백엔드 퀵스타트 문서화

현재 판단:
- 개발/데모 가능한 형태의 수직 슬라이스(앱->API->DB)는 확보됨
- 운영 안정화와 품질 고도화 단계로 넘어갈 수 있는 상태

### 2-2. 백엔드(FastAPI)

구현된 주요 API:
- `GET /health`
- `GET /search?q=&limit=`
- `GET /cases/{case_number}`
- `GET /cases?limit=`
- `GET /dashboard/recommended?limit=`
- `GET /dashboard/wrong-answers?user_id=&limit=`

완료 내용:
- 사건번호/키워드 기반 검색 API 구성
- 상세 조회 API 구성
- 대시보드 추천/오답 목록 API 추가
- pydantic 응답 스키마 정리

현재 제한:
- 추천 로직이 간단한 점수식 기반(고도화 여지 큼)
- 인증/권한/사용자별 세션 관리는 본격 도입 전 단계
- 검색 정확도 고도화(재순위화, 쿼리 확장 등)는 미완료

### 2-3. 데이터/DB(PostgreSQL)

완료 내용:
- `schema.sql` 기반 초기화 체계 확보
- 운영 데이터 적재 기준으로 DB 조회 체계 정비
- 사용자별 `user_case_history` 기반 조회 체계 정비
- 게시 상태 뷰(`published_cases`)를 기준으로 API 조회되도록 정리

운영 규칙 측면 진행:
- `raw/normalized/reviewed/failed/manifests` 데이터 단계 구조 정의
- 정책 체크/메일 템플릿/manifest 템플릿 등 데이터 운영 문서 확보

현재 제한:
- 실제 대규모 최신 판례 배치 파이프라인 자동화는 미완료
- 데이터 품질 지표(정확도, 누락률, 최신성) 대시보드는 부재

### 2-4. iOS 앱(SwiftUI)

완료 기능:
- 5탭 구조 구현: Home / OCR / Search / Review / My Page
- Home: 백엔드 대시보드(추천/오답) 로딩 및 실패 폴백 처리
- OCR: PhotosPicker + Vision OCR, 인식 텍스트를 Search 탭으로 전달
- Search: 키워드 검색, 결과 카드, 상세(판례 요약) 진입
- Case Summary: 요약/퀴즈 생성 트리거 및 상태 표시
- Quiz: 객관식 표시/정답 확인/해설/오답 저장 흐름
- Review: 오답 리스트 조회
- My Page: API 서버 주소 오버라이드 저장/초기화
- 주요 화면 공통 소형 뒤로가기 UX 반영

현재 제한:
- 네트워크 실패/시간초과/오프라인 케이스 UX를 더 세분화할 필요
- 오답 저장의 서버 동기화 정책(온라인/오프라인 충돌 처리) 정교화 필요
- 접근성(폰트 스케일, VoiceOver, 대비) 검증 및 개선 필요

### 2-5. 온디바이스 LLM

완료 내용:
- 서버형 LLM 의존 전략에서 온디바이스(로컬) 추론 전략으로 전환
- `LlamaSwift` 연동 경로 확보
- `LLMService` 상태 머신(loading/ready/inferring/error) 적용
- 요약/퀴즈/비교 프롬프트 흐름 구현
- 엔진 실패 시 Rule-based fallback으로 UX 보호
- 앱 시작 시 선로딩 및 ready 대기 로직 반영

전략 전환 배경(서버형 -> 로컬형):
- 변경 전: 서버 API 호출 기반 생성형 응답 확장 가능성을 우선 검토
- 변경 후: 실기기 학습 환경에서 네트워크 의존도를 낮추기 위해 로컬 추론을 기본 전략으로 채택
- 전환 이유:
  - 오프라인/저연결 환경에서도 핵심 학습 흐름 유지
  - 민감 학습 데이터의 외부 전송 최소화
  - 서버 비용/지연시간 변동 리스크 완화
- 적용 방식:
  - 1차 생성 경로는 로컬 엔진(LlamaSwift + GGUF)
  - 초기화/추론 실패 시 규칙 기반 fallback으로 기능 연속성 확보

현재 제한:
- 출력 포맷 파싱 안정성 개선 필요
- 모델/샘플링 파라미터 튜닝 미완료
- 실기기 성능(지연/발열/메모리) 계측 기반 최적화 미흡

### 2-6. 테스트/품질

완료:
- iOS 테스트 실행 명령 및 시뮬레이터/캐털리스트 실행 경로 문서화
- 백엔드 상태/조회 API 수동 검증 루틴 확보

부족:
- 백엔드 자동 테스트(단위/통합) 부족
- iOS UI 테스트/E2E 자동화 부족
- 회귀 테스트 체크리스트 체계화 필요

## 3. 현재 단계 진단

단계 정의상 현재 위치:
- "핵심 기능 1차 구현 완료 + E2E 안정화/품질 고도화 착수 직전"

의미:
- 기능 데모는 가능한 상태
- 실사용 수준으로 가기 위해 안정성/정확도/운영성 작업이 필수

핵심 리스크 3가지:
1. LLM 출력 일관성 부족으로 사용자 신뢰 저하 가능
2. 추천/검색 품질 고도화 미흡 시 학습 효율 가치 약화
3. 자동 테스트 부족으로 기능 추가 시 회귀 위험 증가

## 4. 앞으로 해야 할 일 (우선순위 포함)

## 4-1. 최우선 (P0: 즉시 착수)

1. 실기기 E2E 검증 루프 고정
- 목표: iPhone에서 Home/OCR/Search/요약/퀴즈/오답 저장 전 경로 안정 동작
- 작업:
  - 네트워크 주소 오버라이드 시나리오 재검증
  - 앱 백그라운드 복귀/네트워크 끊김 시나리오 점검
  - 실패 케이스별 사용자 메시지 통일
- 완료 기준:
  - 치명 오류(크래시/무응답) 0건
  - 주요 경로 성공률 95% 이상(내부 테스트 기준)

2. LLM 출력 안정화
- 목표: 요약/퀴즈 파싱 실패율 유의미하게 감소
- 작업:
  - 프롬프트 포맷 고정(필드 키 강제)
  - 파서 보강(누락 필드 보정, 안전 디폴트)
  - 실패 로그 수집 포인트 추가
- 완료 기준:
  - 파싱 실패율 목표치 설정 및 달성(예: < 5%)

3. 추천/검색 품질 1차 고도화
- 목표: 사용자 체감 정확도 개선
- 작업:
  - 추천 점수식 개선(최근 학습/오답 가중치 반영)
  - 검색 쿼리 정규화(불용어/동의어/사건번호 우선순위)
- 완료 기준:
  - 내부 샘플 질의셋에서 Top-N 적합률 개선 확인

## 4-2. 중요 (P1: 단기 1~2주)

1. 테스트 체계 구축

3. 관측성/로그 개선

## 9. 진행 중 변경 사안 정리 (초기 계획 대비)

### 9-1. 아키텍처/운영 변경

- 변경 전: 문서상 API/DB 실행 경로는 있었지만, compose 프로젝트 혼선 이슈가 운영 리스크로 존재
- 변경 후: compose 스택을 AI_SYS_TEAM 기준으로 정리하고, 404 발생 원인이던 잘못된 프로젝트 점유 상태를 해소
- 영향: `/dashboard/recommended` 등 핵심 API 응답 안정성 개선, 데모/개발 재현성 향상

### 9-2. 데이터 소스 변경 (앱 더미 -> DB 중심)

- 변경 전: iOS 화면 흐름 일부가 Swift 더미 데이터 중심으로 동작
- 변경 후: 실데이터 및 `user_case_history` 기반으로 API 조회
- 영향: 앱-API-DB 데이터 정합성 개선, 실제 운영 형태와 유사한 검증 가능

### 9-3. 대시보드 범위 확장

- 변경 전: 기본 케이스 조회/검색 API 중심
- 변경 후: `GET /dashboard/recommended`, `GET /dashboard/wrong-answers` 추가
- 영향: Home 화면이 단순 정적 카드가 아닌 학습 우선순위/오답 복습 중심 화면으로 전환

### 9-4. iOS 네트워크 설정 전략 변경

- 변경 전: 고정 API 주소 기반 사용 비중이 높아 실기기 테스트 시 네트워크 전환이 번거로움
- 변경 후: My Page에서 API 주소 오버라이드 저장/초기화 기능 도입
- 영향: 실기기 환경(Mac 로컬 서버, 네트워크 변경)에서 테스트 전환 속도 향상

### 9-5. OCR 연계 강화

- 변경 전: OCR은 기능 검증 수준, 검색 흐름 연결이 느슨함
- 변경 후: OCR 인식 결과를 정제해 Search 탭 쿼리로 자동 전달하는 런타임 상태 연동 적용
- 영향: 사용자 학습 흐름이 "스캔 -> 검색 -> 요약"으로 단절 없이 연결됨

### 9-6. LLM 전략 변경 (안정성 우선)

- 변경 전: 로컬 LLM 연결은 되었으나 초기 not-ready 상황에서 사용자 경험 저하 가능
- 변경 후:
  - 서버형 LLM 의존 방향에서 로컬 추론 우선 전략으로 기준 전환
  - 앱 시작 시 선로딩
  - 요약/퀴즈 실행 전 ready 대기 로직 추가
  - 실패 시 Rule-based fallback 유지
- 영향: LLM 초기화 구간 오류 노출 감소, 실패 시에도 핵심 학습 흐름 유지

추가 전략 메모:
- 단기 운영 기준: 로컬 추론을 기본 경로로 유지
- 예외 경로: 로컬 성능/품질 임계치 미달 시 서버 보조 경로를 옵션으로 검토
- 의사결정 기준: 지연시간, 배터리/발열, 파싱 실패율, 사용자 체감 정확도
